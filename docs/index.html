The problem

Agents act.
You find out later.

Modern agent frameworks make it easy to build agents, but hard to control them once they start acting autonomously.

🔧

Unexpected tool calls

Agents reach tools you didn't intend, with no gate between them and the action.

💸

Hidden token spend

LLM cost grows through loops and retries with no per-agent budget enforcement.

🔍

Blind MCP traffic

MCP servers run on different hosts with no central record of what was called or by whom.

🔗

Uncontrolled A2A

Agent-to-agent calls bypass central security logic with no identity verification.

🧾

Traces without decisions

Traces show what happened, but not why it was allowed — no policy context in the record.

Architecture

The proxy is
the control plane

aMaze intercepts all agent traffic at the network layer. Point HTTP_PROXY at it — no agent code changes, no framework integration required.

Agent ──── HTTP_PROXY ────► aMaze Proxy (mitmproxy + addon chain) │ ┌───────────────────────┼───────────────────────┐ ▼ ▼ ▼ LLM Provider MCP Server Peer Agent openai, anthropic… tool calls A2A calls

Every request passes through the addon chain in order. If any step raises, the request is denied. There is no fail-open mode.

session_id
Resolve bearer token → agent identity. Strip client-supplied identity headers, inject trusted x-amaze-caller.
↓
policy_enforcer
A2A / MCP / LLM allowlist checks against the agent's policy in Redis.
↓
graph_enforcer
Step ordering and loop limits — strict or flexible execution graph mode.
↓
stream_blocker
Inject "stream": false into LLM requests — guarantees complete audit records and accurate token counting.
↓
tracer
Open OTel span on request, close on response. All other addons attach attributes to the same span.
↓
router
Resolve host → upstream URL from Redis, forward the request.
↓
counters
RedisTimeSeries — token and call metrics per agent, used for time-window rate limit pre-checks.
↓
audit_log
XADD to Redis Stream — one immutable record per call, linked to the OTel span via trace_id.
Capabilities

Everything you need
to govern agents

One YAML policy per agent, stored in Redis at runtime. Changes via the GUI or API take effect on the next request — no restart.

🔐

A2A enforcement

Control which agents can call which peers. Identity is injected by the proxy — never client-controlled.

🔧

MCP tool enforcement

Allow or deny specific tools per MCP server. Unknown servers and tools are blocked by default.

🤖

LLM enforcement

Allowlist providers and models. Cap token spend per turn and per time window via RedisTimeSeries.

📊

Execution graph

Strict mode: enforce exact call ordering with per-step loop limits. Flexible mode: any-order allowed set.

⏱️

Per-turn limits

max_tokens, max_tool_calls, max_agent_calls — reset at the start of each user message.

📋

Immutable audit log

Redis Streams — every call, allowed or denied, with full input/output, denial reason, and OTel link.

🔭

Distributed traces

OTel spans → Jaeger. Each conversation shares one trace_id. A2A hops propagate trace context.

🖥️

GUI dashboard

Live KPIs, sequence diagrams, alert views, and a drag-to-connect execution graph editor.

🔌

Zero framework coupling

LangChain, CrewAI, raw httpx — anything that sets HTTP_PROXY works without modification.

Why aMaze

not inside agent code

Quick start

Up and running
in minutes

Prerequisites: Docker + Docker Compose. See examples/compose.yml for a complete multi-agent reference.

1

Start the control plane

bash
$ docker compose -f docker/docker-compose.yml up -d
2

Wire aMaze into your agent

my_agent.py
import amaze # Define one or both handlers — aMaze discovers them automatically. async def receive_message_from_user(message): result = await my_agent.ainvoke({"messages": [{"role": "user", "content": message}]}) return result["messages"][-1].content async def receive_message_from_agent(caller, message): return await receive_message_from_user(message) if __name__ == "__main__": amaze.init() # reads AMAZE_AGENT_ID from env, registers, starts servers
3

Add your agent as a compose service

docker-compose.agents.yml
services: my-agent: build: . environment: AMAZE_AGENT_ID: my-agent AMAZE_ORCHESTRATOR_URL: http://host.docker.internal:8001 HTTP_PROXY: http://host.docker.internal:8080 HTTPS_PROXY: http://host.docker.internal:8080 extra_hosts: - "host.docker.internal:host-gateway"

bash
$ docker compose \ -f docker/docker-compose.yml \ -f docker-compose.agents.yml \ up --build
✅ Your agent is now under full policy enforcement. Jaeger traces at http://localhost:16686 · Control plane GUI at http://localhost:5173
Policy model

One YAML policy
per agent

Stored in Redis at runtime. The GUI and PUT /policy/{id} API write to Redis directly — changes are live on the next request, no restart needed.

config/policies.yaml
name: my-agent-policy # Per-turn limits — reset at the start of each user message max_tokens_per_turn: 10000 max_tool_calls_per_turn: 20 max_agent_calls_per_turn: 5 allowed_llm_providers: [openai, anthropic] # Time-window token budgets via RedisTimeSeries token_rate_limits: - window: 10m max_tokens: 2000 - window: 1h max_tokens: 10000 # alert always written; these control blocking only on_budget_exceeded: block # block | allow on_violation: block # block | allow # flexible: any-order allowed set mode: flexible allowed_tools: [web_search, summarize] allowed_agents: [summarizer-agent] # strict: enforce exact call order with per-step loop limits # mode: strict # graph: # start_step: 1 # steps: # - step_id: 1 # call_type: tool # callee_id: web_search # max_loops: 3 # next_steps: [2] # - step_id: 2 # call_type: agent # callee_id: summarizer-agent # max_loops: 1 # next_steps: []

Failure handling

Fails closed.
Always.

If any addon in the chain raises, the request is denied. There is no fail-open path.

ConditionResult
Unknown bearer token 403 invalid-bearer
Agent not in peer's allowlist 403 not-allowed
MCP server not registered 403 mcp-not-allowed
Tool not in policy 403 tool-not-allowed
LLM provider / model denied 403 llm-not-allowed
Graph step out of order 403 graph_violation
Loop limit exceeded 429 edge_loop_exceeded
Token budget exceeded 403 budget_exceeded
Rate limit exceeded 403 rate-limit-exceeded
Redis unavailable 503 (fail closed)
Any addon raises deny (fail closed)
Deployment

Two containers.
Full stack.

The platform container runs the proxy, orchestrator, and Jaeger under supervisord. Redis runs in a sidecar. Remote agents connect over HTTP_PROXY — no shared Docker network required.

ProcessContainerPortRole
redis-stack amaze-redis 6379 State, counters, streams, timeseries
orchestrator amaze-platform 8001 Session lifecycle, policy CRUD, audit API
proxy amaze-platform 8080 mitmproxy + enforcement addon chain
jaeger amaze-platform 16686 / 4317 OTel trace storage + UI
ui amaze-ui 5173 React control dashboard
